Introduction

The emergence of large language models (LLMs) as the foundation of production software has created a new class of infrastructure requirement: AI application memory. Unlike traditional databases optimised for structured data retrieval, AI applications require (i) semantic search by meaning rather than exact match, (ii) temporal relevance that weights recent or frequently confirmed knowledge higher, (iii) relational context to follow chains of related knowledge, and (iv) operational simplicity for edge, serverless, or single-file deployments.

Existing approaches partition these concerns across multiple systems—a vector database  for semantic search, a key-value store for recency metadata, and a graph database for relational context—introducing synchronisation complexity, inter-process latency, and operational overhead.

We argue these concerns are not separable: they are facets of a single unified problem—how does an AI application recall the right context at the right time?—and address it with a single embedded system.

Contributions.

  1. Adaptive Temporal Decay (ATD): A scoring formula coupling time-decay with access-frequency stickiness, preventing important memories from fading while dormant knowledge recedes naturally.

  2. Embedded Hybrid Retrieval: BM25 inverted index and HNSW dense vector index fused via RRF inside a single process, with no external synchronisation, query planning, or schema definition.

  3. Context Chain: A single-call operation combining vector ANN search with typed graph BFS expansion for multi-hop relational context retrieval.

  4. Production Hardening for Embedded AI: Write-ahead log, atomic saves, per-instance mutex, and soft-delete with index compaction.

Background and Related Work

Vector Databases

FAISS  provides highly optimised ANNS indices as a library but offers no persistence, metadata, or query composability. Chroma , Qdrant , and Weaviate  build on similar indices with REST APIs and persistence but operate as server processes. None implement temporal scoring as a first-class primitive.

HNSW

Hierarchical Navigable Small World graphs  provide O(log⁡n)O(\log n) query complexity with high recall by constructing a multi-layer proximity graph traversed from coarse to fine resolution. We use the hnswlib implementation  with M=16M=16, ef_construction=200ef\_\text{construction}=200.

BM25

Okapi BM25  is a probabilistic term-frequency ranking function: BM25(d,Q)=∑qi∈QIDF(qi)⋅f(qi,d)(k1+1)f(qi,d)+k1(1−b+b|d|dl¯)\begin{equation} \text{BM25}(d,Q) = \sum_{q_i \in Q} \text{IDF}(q_i) \cdot \frac{f(q_i,d)(k_1+1)}{f(q_i,d)+k_1\!\left(1-b+b\frac{|d|}{\overline{dl}}\right)} \label{eq:bm25} \end{equation} where k1=1.2k_1=1.2, b=0.75b=0.75, and IDF(qi)=log⁡(N−nt+0.5nt+0.5+1)\text{IDF}(q_i)=\log\!\left(\frac{N-n_t+0.5}{n_t+0.5}+1\right).

Reciprocal Rank Fusion

RRF  is a rank-fusion method robust to score-scale differences: RRF(d)=∑r1krrf+rankr(d)+1\begin{equation} \text{RRF}(d) = \sum_r \frac{1}{k_{\text{rrf}} + \text{rank}_r(d) + 1} \label{eq:rrf} \end{equation} with krrf=60k_{\text{rrf}}=60 (default).

Temporal Decay

Time-decay scoring typically models relevance as an exponential function of age . MemGPT  and Generative Agents  apply fixed exponential decay to agent memory. Our work extends this with a stickiness factor derived from access frequency, analogous to the spacing effect in human memory consolidation .

Graph-Enhanced Retrieval

GraphRAG  augments LLM retrieval with graph traversal over entity relationship graphs using separate graph databases. Feather DB integrates graph traversal into the vector search index, eliminating the separate graph store.

System Design

Data Model

A Feather DB instance manages three in-memory structures:

  • Metadata Store: unordered_map<uint64_t, Metadata>—keyed by entity ID, carrying timestamp, importance, content, type, namespace, entity ID, typed edges, recall count, and a free-form attribute map.

  • Modality Indices: unordered_map<string, ModalityIndex>—named HNSW indices (e.g., “text”, “visual”), created on demand with independent dimensionality.

  • BM25 Index: unordered_map<string, vector<PostingEntry>>—inverted index over content tokens, rebuilt from the metadata store on every open().

Adaptive Temporal Decay (ATD)

Let rr be a record with recall count cc, creation timestamp tt, and importance weight α∈[0,1]\alpha \in [0,1]. For a query at time τ\tau with half-life HH and blend weight ww:

stickiness(c)=1+log⁡(1+c)age_days=(τ−t)/86400eff_age=age_days/stickiness(c)recency(r)=0.5eff_age/Hscore(r,q)=[(1−w)⋅sim(r,q)+w⋅recency(r)]⋅α\begin{align} \text{stickiness}(c) &= 1 + \log(1 + c) \label{eq:stickiness}\\ \text{age\_days} &= (\tau - t) / 86400\\ \text{eff\_age} &= \text{age\_days} / \text{stickiness}(c)\\ \text{recency}(r) &= 0.5^{\,\text{eff\_age} / H} \label{eq:recency}\\ \text{score}(r,q) &= \bigl[(1-w)\cdot\text{sim}(r,q) + w\cdot\text{recency}(r)\bigr]\cdot\alpha \label{eq:score} \end{align}

Theorem 1 (Asymptotic Stickiness). As c→∞c \to \infty, stickiness(c)→∞\text{stickiness}(c) \to \infty, eff_age→0\text{eff\_age} \to 0, and recency(r)→1\text{recency}(r) \to 1 regardless of actual age. A memory accessed infinitely often never decays.

Theorem 2 (Neutral Baseline). For c=0c=0, stickiness=1\text{stickiness}=1 and the formula reduces to standard exponential decay: recency=0.5age/H\text{recency} = 0.5^{\,\text{age}/H}.

The logarithmic form of stickiness is deliberately sub-linear, rewarding genuinely important long-term memories without allowing trivially-recalled records to dominate.

Tokenisation.

Lowercase, split on non-alphanumeric characters, minimum token length 2, 40 common English stop-words filtered.

Incremental maintenance.

Document lengths are tracked per-record. The corpus average document length dl¯\overline{dl} is updated incrementally: dl¯(N)=dl¯(N−1)(N−1)+|d|N\begin{equation} \overline{dl}^{(N)} = \frac{\overline{dl}^{(N-1)}(N-1) + |d|}{N} \end{equation} avoiding a full corpus rescan on each insertion.

Hybrid Search Pipeline

Input: query_vec q, query_string s, k
1. HNSW search(q, 4k)  -> vec_results
2. BM25 search(s, 4k)  -> kw_results
3. RRF(vec_results, kw_results, rrf_k=60)
4. touch_nolock(id) for top-k
5. Return top-k by RRF score

Steps 1–5 execute under a single std::lock_guard to prevent deadlock. The 4k4k over-fetch ensures sufficient candidates for RRF to produce meaningful top-kk results even when both arms have sparse overlap.

Context Chain Traversal

Input: query_vec q, k seeds, hops h
1. HNSW search(q, k)  -> seed_nodes (hop=0)
2. BFS expansion:
   for hop in 1..h:
     for node at hop-1:
       for edge e of node:
         if target unseen:
           score = cosine(target_vec, q)
           add to results at hop distance
3. Return ContextChainResult{nodes, edges}

Similarity scores for graph-expanded nodes (hop>0\text{hop} > 0) are computed as cosine similarity to the query vector, maintaining retrieval-grounded scores even for nodes reached via graph traversal.

Write-Ahead Log and Atomic Saves

Every write operation appends to {path}.wal:

[op_code: 1B] [payload_len: 4B] [payload: N B]

On open(), after loading the main .feather file, the WAL is replayed if present. On save(), the main file is written atomically via POSIX rename() and the WAL is cleared.

Proposition 1 (Crash Safety). Any write confirmed to the application (add() returned) is recoverable after a crash, subject to OS filesystem sync guarantees.

Concurrency

Feather DB uses a single std::mutex per DB instance with std::lock_guard on all public methods. A private touch_nolock() method allows search(), keyword_search(), and context_chain() to update recall counts without re-acquiring the lock. hybrid_search() inlines both retrieval arms under one lock to prevent recursive deadlock.

Dataset: 3,050 synthetic records, 256-dimensional unit-normalised vectors.
Topics: 10 semantic clusters (billing, API usage, security, performance, etc.).

Limitations and Future Work

Cold open latency. For >100> 100k records, HNSW reconstruction and BM25 rebuild during open() will be slow. Planned fix: persist the BM25 posting list in the .feather file.

Single-writer concurrency. The coarse mutex prevents concurrent writes. Future work will explore reader-writer locks for read-heavy workloads.

Synthetic benchmarks. Evaluation uses clustered synthetic embeddings. Future work will benchmark with sentence-transformers (MiniLM, BGE) and OpenAI embeddings on standard IR benchmarks (BEIR , MS-MARCO).

No query language. Future work: a minimal declarative DSL over the filter and graph primitives.

Conclusion

We presented Feather DB, an embedded vector database and living context engine that addresses AI application memory through three integrated mechanisms: adaptive temporal decay, embedded hybrid retrieval, and context chain traversal. Our evaluation demonstrates 0.100.10 ms search latency, production-grade concurrency (15,11915{,}119 q/s), 100% crash recovery, and a 55%55\% relative precision improvement of hybrid retrieval over keyword-only—within a zero-configuration embedded deployment model with a <4<4 MB file footprint for 3,000 records.

Feather DB makes the case that AI application memory is not a storage problem but a retrieval relevance problem. By treating temporal adaptation, graph relations, and semantic similarity as first-class primitives rather than application-layer concerns, Feather DB provides a foundation on which AI applications can build persistent, intelligent, and adaptive memory at any scale.
