Abstract

AI-native applications require memory systems that go beyond static vector retrieval: they need context that is temporally adaptive, relationally connected, and queryable via both semantic similarity and exact keyword matching. We present Feather DB, an embedded vector database and living context engine that introduces three novel contributions: (1) Adaptive Temporal Decay (ATD)—a scoring formula that modulates time-based relevance decay using access frequency, preventing frequently recalled memories from fading; (2) embedded hybrid retrieval—tight integration of Okapi BM25 and HNSW approximate nearest-neighbour search fused via Reciprocal Rank Fusion (RRF), with zero index synchronisation overhead; and (3) context chain traversal—a single-call operation combining semantic vector search with $n$-hop graph BFS expansion over a typed, weighted knowledge graph, enabling relational context retrieval in a single call. We evaluate Feather DB on a synthetic multi-tenant AI memory workload of 3,050 records across 10 topic categories, demonstrating $0.10$ ms vector search latency, $15{,}119$ concurrent queries/sec across 8 threads, 100% write-ahead-log crash recovery, and 70% hybrid search topic precision versus 44% for keyword-only retrieval. Feather DB is open-source (MIT) and available at github.com/feather-store/feather.

Introduction

The emergence of large language models (LLMs) as the foundation of production software has created a new class of infrastructure requirement: AI application memory. Unlike traditional databases optimised for structured data retrieval, AI applications require (i) semantic search by meaning, (ii) temporal relevance that weights frequently confirmed knowledge higher, (iii) relational context to follow chains of related knowledge, and (iv) operational simplicity for edge, serverless, or single-file deployments.

Existing approaches partition these concerns across multiple systems—a vector database [1,2,3] for semantic search, a key-value store for recency metadata, and a graph database for relational context— introducing synchronisation complexity, inter-process latency, and operational overhead.

We argue these concerns are not separable: they are facets of a single unified problem—how does an AI application recall the right context at the right time?—and address it with a single embedded system.

Contributions.

  1. Adaptive Temporal Decay (ATD): A scoring formula coupling time-decay with access-frequency stickiness, preventing important memories from fading while dormant knowledge recedes.
  2. Embedded Hybrid Retrieval: BM25 inverted index and HNSW dense vector index fused via RRF inside a single process, with no external synchronisation or schema definition.
  3. Context Chain: A single-call operation combining vector ANN search with typed graph BFS expansion for multi-hop relational context retrieval.
  4. Production Hardening: Write-ahead log, atomic saves, per-instance mutex, and soft-delete with index compaction.

Background and Related Work

Vector Databases

FAISS [4] provides highly optimised ANNS indices as a library but offers no persistence, metadata, or query composability. Chroma [3], Qdrant [5], and Weaviate [2] build on similar indices with REST APIs and persistence but operate as server processes. None implement temporal scoring as a first-class primitive.

HNSW

Hierarchical Navigable Small World graphs [6] provide $O(\log n)$ query complexity with high recall by constructing a multi-layer proximity graph traversed from coarse to fine resolution. We use the hnswlib implementation [7] with $M{=}16$, $ef_\text{construction}{=}200$.

BM25

For query $Q = \{q_1,\ldots,q_n\}$ and document $d$, Okapi BM25 [8] scores:

$$\text{BM25}(d,Q) = \sum_{q_i \in Q} \text{IDF}(q_i) \cdot \frac{f(q_i,d)(k_1+1)}{f(q_i,d)+k_1\!\left(1-b+b\tfrac{|d|}{\bar{dl}}\right)}$$

where $k_1{=}1.2$, $b{=}0.75$, and $\text{IDF}(q_i)=\log\!\left(\tfrac{N-n_t+0.5}{n_t+0.5}+1\right)$.

Reciprocal Rank Fusion

RRF [9] merges heterogeneous ranked lists without score normalisation:

$$\text{RRF}(d) = \sum_r \frac{1}{k_\text{rrf} + \text{rank}_r(d) + 1}$$

with $k_\text{rrf}{=}60$. Its rank-based nature makes it robust to scale differences between BM25 and cosine similarity.

Temporal Decay

Time-decay scoring models relevance as an exponential function of age [10]. MemGPT [11] and Generative Agents [12] apply fixed exponential decay to agent memory. Our work extends this with a stickiness factor derived from access frequency, analogous to the spacing effect in human memory consolidation [13].

Graph-Enhanced Retrieval

GraphRAG [14] augments LLM retrieval with graph traversal over entity relationship graphs using separate graph databases. Feather DB integrates graph traversal into the vector search index, eliminating the separate graph store.

System Design

Data Model

A Feather DB instance manages three in-memory structures:

  • Metadata Store: unordered_map<uint64_t, Metadata>—keyed by entity ID, carrying timestamp, importance, content, type, namespace, entity ID, typed edges, recall count, and a free-form attribute map.
  • Modality Indices: unordered_map<string, ModalityIndex>—named HNSW indices (e.g., "text", "visual"), created on demand with independent dimensionality.
  • BM25 Index: Inverted posting list over content tokens, rebuilt from the metadata store on every open().

Adaptive Temporal Decay

Let $r$ be a record with recall count $c$, creation timestamp $t$, and importance $\alpha \in [0,1]$. For a query at time $\tau$ with half-life $H$ and blend weight $w$:

$$\text{stickiness}(c) = 1 + \log(1 + c) \tag{1}$$ $$\text{eff\_age} = \frac{(\tau-t)/86400}{\text{stickiness}(c)} \tag{2}$$ $$\text{recency}(r) = 0.5^{\,\text{eff\_age}/H} \tag{3}$$ $$\text{score}(r,q) = \bigl[(1-w)\cdot\text{sim}(r,q) + w\cdot\text{recency}(r)\bigr]\cdot\alpha \tag{4}$$
Theorem 1 (Asymptotic Stickiness). As $c \to \infty$, $\text{stickiness}(c) \to \infty$, $\text{eff\_age} \to 0$, and $\text{recency}(r) \to 1$ regardless of actual age. A memory accessed infinitely often never decays.
Theorem 2 (Neutral Baseline). For $c{=}0$, stickiness$=1$ and the formula reduces to standard exponential decay: $\text{recency} = 0.5^{\,\text{age}/H}$.

BM25 Inverted Index

Tokenisation. Lowercase, split on non-alphanumeric characters, minimum token length 2, 40 common English stop-words filtered.

Incremental maintenance. The corpus average document length $\bar{dl}$ is updated incrementally on each add(), avoiding a full corpus rescan: $$\bar{dl}^{(N)} = \frac{\bar{dl}^{(N-1)}(N-1) + |d|}{N} \tag{5}$$

Hybrid Search Pipeline

Listing 1: Hybrid search pseudocode
Input: query_vec q, query_string s, k
1. HNSW search(q, 4k)  → vec_results
2. BM25 search(s, 4k)  → kw_results
3. RRF(vec_results, kw_results, rrf_k=60)
4. touch_nolock(id) for top-k
5. Return top-k by RRF score

Steps 1–5 execute under a single std::lock_guard to prevent deadlock. The $4k$ over-fetch ensures sufficient candidates for RRF even when both arms have sparse overlap.

Context Chain Traversal

Listing 2: Context chain pseudocode
Input: query_vec q, k seeds, hops h
1. HNSW search(q, k)   → seed_nodes (hop=0)
2. BFS expansion:
   for hop in 1..h:
     for node at hop-1:
       for edge e of node:
         if target unseen:
           score = cosine(target_vec, q)
           add at hop distance
3. Return {nodes, edges}

Write-Ahead Log and Atomic Saves

Every write appends to {path}.wal:

[op_code: 1B] [payload_len: 4B] [payload: N B]

On open(), the WAL is replayed after loading the main file. On save(), the main file is written atomically via POSIX rename() and the WAL is cleared.

Proposition 1 (Crash Safety). Any write confirmed to the application (add() returned) is recoverable after a crash, subject to OS filesystem sync guarantees.

Concurrency Model

Feather DB uses a single std::mutex per DB instance with std::lock_guard on all public methods. A private touch_nolock() method allows search() and keyword_search() to update recall counts without re-acquiring the lock. hybrid_search() inlines both retrieval arms under one lock to prevent recursive deadlock.

Evaluation

Experimental Setup

Hardware: Apple M-series (ARM64), 16 GB RAM. Dataset: 3,050 synthetic records, 256-dimensional unit-normalised vectors. Topics: 10 semantic clusters (billing, API usage, security, performance, user prefs, etc.). Namespaces: 3 tenants. Embeddings: Clustered synthetic vectors (centroid + Gaussian noise $\sigma{=}0.08$, unit-normalised). Known cluster membership enables precise precision measurement without dependence on a specific embedding model.

Adaptive Decay Validation

A record was touched $c{=}20$ times before search. With $H{=}30$ days, $w{=}0.3$: $$\text{stickiness}(20) = 1 + \log(21) \approx 4.04$$ For a 7-day-old record, effective age drops from $7.0$ to $1.73$ days. Recency improves from $0.857$ to $0.961$—a 12% improvement that competes with similarity for rank promotion.

Limitations and Future Work

Cold open latency. For $>100$k records, HNSW reconstruction and BM25 rebuild will be slow. Planned fix: persist the BM25 posting list directly in the .feather file.

Single-writer concurrency. The coarse mutex prevents concurrent writes. Future work will explore reader-writer locks for read-heavy workloads, and optimistic concurrency for write-heavy ones.

Synthetic benchmarks. Future work will benchmark with sentence-transformers (MiniLM, BGE) and OpenAI embeddings on BEIR [15] and MS-MARCO to measure real-world retrieval quality.

No query language. Future work: a minimal declarative DSL over the filter and graph primitives for complex multi-hop joins.

Conclusion

We presented Feather DB, an embedded vector database and living context engine that addresses AI application memory through three integrated mechanisms: adaptive temporal decay, embedded hybrid retrieval, and context chain traversal. Our evaluation demonstrates $0.10$ ms search latency, production-grade concurrency ($15{,}119$ q/s), 100% crash recovery, and a 55% relative precision improvement of hybrid over keyword-only retrieval—within a zero-configuration embedded deployment with a $<4$ MB file footprint for 3,000 records.

Feather DB makes the case that AI application memory is not a storage problem but a retrieval relevance problem. By treating temporal adaptation, graph relations, and semantic similarity as first-class primitives, Feather DB provides a foundation on which AI applications can build persistent, intelligent, and adaptive memory at any scale.
